# sitecue フォント設計ポリシー

sitecueでは、「引き算の美学」および「極限のパフォーマンスと執筆UXの両立」を目的として、意図的にアプリ全体とエディタ部分でフォントの扱いを分けています。

## 1. フォント適用ルール

| 領域 | 適用フォント | 読み込み方法 | 目的 |
| :--- | :--- | :--- | :--- |
| **アプリ全体 (UI)** | **OS標準フォント** (System Fonts) | なし（ネイティブ） | ゼロ・ペイロードによる最速の描画と、OSに馴染む自然な操作感の提供。 |
| **エディタ領域**<br>*(Notes / Studio)* | **Hack (Subset)** + OS標準 | `next/font/local` | 英語・Markdown記号の視認性向上と、日本語フォールバックによる軽量化の両立。 |

## 2. アーキテクチャの背景と理由

### アプリ全体にWebフォントを適用しない理由
日本語を含むWebフォント（Noto Sans JPなど）は、ファイルサイズが数MBに及ぶことがあり、初回ロード時のレイアウトシフト（FOUT）やパフォーマンス低下の大きな原因となります。
sitecueは「軽快な動作」をプロダクトのコア価値としているため、UIコンポーネント（ボタン、ナビゲーション、ダッシュボード等）には不要なフォントデータを一切読み込ませず、ユーザーのOS標準フォント（游ゴシック、ヒラギノ、SF Pro等）に依存させるアプローチを採用しています。

### エディタ領域のみに「Hack（サブセット版）」を導入する理由
コードやMarkdownの執筆体験を向上させるためには、等幅で記号（`#`, `>`, `*`, `-` 等）の視認性が高いフォントが不可欠です。
ここでもパフォーマンスを犠牲にしないため、以下の戦略をとっています。

1. **サブセット化による極小ペイロード:**
   世界中の文字を含むフルバージョンではなく、英数字と主要な記号のみを残した「サブセット版（十数KB程度）」のHackフォントのみをセルフホストして読み込んでいます。
2. **日本語の自然なフォールバック:**
   `font-family: var(--font-hack), system-ui, sans-serif;` と指定することで、「Hackに存在しない日本語文字」は自動的にOS標準の綺麗なフォントへとフォールバックされます。

これにより、「英語・記号は美しいコーディングフォント」「日本語はOS標準フォント」という理想的な混植環境を、ほぼゼロに近い通信コストで実現しています。
