# Webダッシュボード（Next.js）の拡張ルール

## 1. 独立ルーティング（隔離）の原則

Webダッシュボード（`web/`）に新しい機能や画面を追加する場合、既存のページ（`/app/page.tsx` など）のUIを直接改修して複雑化させることは厳禁です。
必ず Next.js の App Router の仕組みを利用し、**完全に独立した専用ページ（特設ルーム）** として作成してください。

- **Bad:** 既存の `web/src/app/page.tsx` の中に、モーダルや条件分岐で新しいAI機能をねじ込む。
- **Good:** `web/src/app/weave/page.tsx` のような新しいルーティングを作成し、既存のコードと物理的に切り離す。

## 2. コンポーネントの隔離

新機能専用のコンポーネントは、共通の `components/` ディレクトリに混ぜるのではなく、機能ごとにディレクトリを切って隔離してください。

- 例: `web/src/components/weave/` 配下にAI機能用のコンポーネントを配置する。
- 理由: 将来的にダッシュボードのトップページを大改修・リファクタリングする際、新機能のコードとコンフリクト（競合）して開発が止まるのを防ぐため。

## 3. アトミックな機能実装

新機能を追加する際は、Web（UI）の追加と API（Hono）の追加を1つのプロンプト・1つのコミットで同時に行わないこと。
まずはUIのモックアップ（箱）だけを独立したルーティング上に作り、その後のステップでAPIとの繋ぎ込みを行う「Expand & Contract パターン」を意識してください。
